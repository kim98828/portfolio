Technical Insights

기술 선택의 이유와 설계 철학

왜 Post-Process가 아닌 엔진 소스를 수정했는가

Post-Process Material로도 셀 셰이딩 효과는 낼 수 있다. 하지만 Deferred Shading의 GBuffer에 커스텀 데이터를 기록할 수 없어, 조명 모델 자체를 NPR로 전환하는 것은 불가능하다고 판단했다.

엔진의 Shading Model을 직접 추가하면 Deferred Lighting 단계에서 NPR BxDF가 동작하여, 모든 라이트 소스에 대해 올바른 셀 셰이딩이 구현되는 편이 낫다고 생각했다. Post-Process 방식 대비 성능 오버헤드 없이 정확한 라이팅 결과를 얻을 수 있을 것이라 판단했다.

Post-Process 화면 공간 기반 → 조명 정보 접근 불가 → 가짜 셀 셰이딩
Engine Source Mod GBuffer 커스텀 → BxDF 레벨 NPR → 정확한 조명 반응

프리뷰 vs 송출 — 프로토콜과 하드웨어의 역할 분리

실시간 방송 시스템에서 "모니터링"과 "본방송 송출"은 요구사항이 다르다고 판단했다. 프리뷰는 유연성과 편의성이 중요하고, 본방송은 프레임 단위의 확실성이 필요하다.

NDI 프리뷰/모니터링 — LAN 자동 디스커버리, 멀티 소스 모자이크, 소프트웨어 기반
DeckLink SDI 본방송 송출 — 하드웨어 확실성, Genlock 프레임 동기화, 방송 표준 충족

NDI는 소프트웨어만으로 동작하여 카메라 전환, 모자이크 프리뷰 등 모니터링 용도에 적합하다고 생각했다. 반면 본방송 송출은 DeckLink SDI 하드웨어를 통해 Genlock 기반 프레임 동기화와 방송 표준을 충족하는 편이 안정적이라 판단했다. 소프트웨어(NDI)와 하드웨어(SDI)의 역할을 분리하면 각각의 강점을 살리면서 시스템 전체의 안정성을 확보할 수 있다.

실시간 스레드 안전 — Lock vs Lock-Free 패턴

UE5의 GameThread와 AnimationThread는 병렬 실행된다. 5인 모캡 데이터를 공유할 때 Mutex를 사용하면 한쪽 스레드가 대기하며 프레임 드롭이 발생할 수 있다고 판단했다.

TSharedPtr의 Atomic Swap을 활용한 스냅샷 패턴이 적합하다고 생각했다. 쓰기 스레드가 새 데이터 객체를 만들고 포인터만 원자적으로 교체하면, 읽기 스레드는 이전 객체를 안전하게 계속 사용한다.

Mutex Lock 스레드 대기 → 16.6ms 예산 초과 → 프레임 드롭
Atomic Swap Lock-Free → 대기 없음 → 60fps 안정 유지

왜 프리컴파일 엔진 배포 전략을 설계했는가

UE5 엔진 소스 빌드는 고사양 PC에서도 40분 이상 소요된다. 캐릭터 모델러, 애니메이터 등 아티스트가 엔진을 직접 빌드하는 것은 비현실적이었다.

해결책으로 역할 기반 배포 전략을 설계했다. 개발자가 커스텀 엔진을 소스에서 빌드하면, 프리컴파일된 바이너리가 SVN NAS로 자동 업로드된다. 아티스트는 SyncPrebuiltEngine 한 번이면 최신 엔진을 받는다.

엔진 소스 수정부에는 // BGRITZ Engine Start YYYY-MM-DD 마커를 의무화하여 커스텀 변경 사항을 추적 가능하게 만들었다. Epic 업스트림 병합 시 충돌 지점을 즉시 파악할 수 있다.

전원 소스 빌드 40분+ 빌드 → 환경 차이 → 빌드 실패 → 아티스트 업무 중단
프리컴파일 배포 개발자 빌드 → NAS 자동 업로드 → 아티스트 원클릭 동기화

AI 코딩 자동화 — Claude Code를 팀 워크플로우에 통합한 이유

3개 레포(Engine/Setup/Project)에 걸친 작업을 혼자서 관리하려면 반복 작업을 극도로 줄여야 했다. CLAUDE.md로 각 레포의 코딩 규칙을 정의하고, 28개 슬래시 커맨드로 반복 워크플로우를 자동화했다.

/engine-modify는 마커 삽입 + 문서 업데이트를 보장하고, /done은 하루의 마무리(문서 정리 → 전체 커밋 → Notion 일일보고)를 원커맨드로 처리한다. 결과적으로 200커밋을 1인이 3개 레포에 걸쳐 일관된 품질로 유지할 수 있었다.

수동 관리 3개 레포 × 코딩 규칙 + 커밋 컨벤션 + 문서 + 보고 = 컨텍스트 스위칭 과부하

왜 모캡 파이프라인을 완전 자동화했는가

5인 동시 모션캡처 환경에서 캐릭터마다 80+ 본 매핑을 수동으로 설정하면, 한 캐릭터당 30분 이상 소요된다. 5인 × 2~3 의상 조합이면 촬영 전 세팅에만 반나절이 걸린다.

MotionBuilder SDK(pyfbsdk)를 활용한 AutoRetarget 자동화를 구축했다. Config 파일에 본 매핑을 정의하고, Vicon 루트 서픽스("_root")와 Rig 캐릭터를 자동 감지하여 캐릭터라이즈 → 리타겟을 원클릭으로 처리한다.

수동 설정 80+ 본 × 5인 × 의상 조합 → 촬영 전 반나절 세팅
AutoRetarget Config 기반 자동 감지 → 5인 리타겟 5분 이내 완료

왜 엔진 빌드에 체크포인트 시스템을 도입했는가

UE5 커스텀 엔진 빌드는 3단계(Setup → GenerateProjectFiles → Build)로 구성되며 전체 소요 시간이 1~3시간이다. 네트워크 끊김이나 디스크 부족 등으로 중간에 실패하면 처음부터 다시 시작해야 했다.

각 단계 완료 시 .manu_build_checkpoint.json에 상태를 기록하는 체크포인트 시스템을 설계했다. 실패 시 마지막 성공 단계부터 재개하고, 전체 빌드 성공 시 체크포인트를 초기화하여 다음 빌드는 클린 상태에서 시작한다.

단일 빌드 3시간 빌드 중 실패 → 전체 재시작 → 개발자 반나절 낭비
체크포인트 단계별 저장 → 실패 지점부터 재개 → 복구 시간 최소화

XROOM — Viewport 오버로드 기반 실시간 컴포지팅

실시간 합성(Compositing)을 위해 Viewport를 오버로드하여 전경·렌더·후경을 레이어별로 분리 렌더링했다. 핵심 문제는 3D Plane에 머티리얼을 매핑하면 해상도 저하가 발생한다는 점이었다.

Plane 크기와 정확히 일치하는 위치에 2D로 직접 드로잉하여 3D 경유 없이 원본 화질을 유지하는 방식으로 해결했다. 이 경험이 DNABLE에서 후경 렌더링에 그대로 이어졌다 — 미디어 텍스처를 정확하고 향상된 품질로 출력하고, 추후 스튜디오 맵에서도 동일한 방식으로 동작하도록 확장 가능한 구조를 미리 준비할 수 있었다.

3D Plane 매핑 텍스처 리샘플링 → 화질 저하 → 해상도 의존적
2D Direct Draw Plane 위치 기반 2D 렌더 → 원본 화질 유지 → 3D 비용 제거

상용·오픈소스 커스텀 엔진들을 분석하여 반음영, 섀도우 채널 분리, 섀도우 블러, 다중 광원, GI, Fake GI를 구현하고, 추가로 아웃라인, 림라이트, 반투명까지 확장했다. 모든 기능을 성능 저하 없이 매핑하여 실제 송출 테스트까지 완전히 마쳤다.

Technical Q&A

실무에서 내린 판단과 그 이유 — 기술 면접 형식의 자문자답

Rendering

UE5에 커스텀 Shading Model을 추가할 때 가장 어려웠던 부분은?

구현 자체는 패턴이 정해져 있다. ShadingCommon.ush에 ID를 등록하고, GBuffer 인코딩을 추가하고, BxDF를 작성하면 된다. BxDF는 하프램버트를 기본으로 설계했고, 하이라이트와 림라이트는 별도 패스에서 연산하는 구조를 전제했다.

핵심 고민은 "어떤 패스를 하이잭할 것인가"라는 아키텍처 의사결정이었다. DeferredLight에서 다중 광원을 이미 구현한 상태였기 때문에, 기존 패스를 건드리면 영향 범위가 넓어진다. 기존 시스템을 유지하면서 NPR을 확장할 수 있는 삽입 지점을 찾는 것 — 그 설계 판단이 가장 중요했다.

Broadcast

NDI 이중 버퍼에서 60fps를 유지하기 위해 뭘 신경 썼나?

먼저 전제 조건을 파악해야 한다. 우리 시스템은 30개 이상의 렌더를 동시에 전송하는 구조이고, 엔진 컴퓨터는 2대였다. 이 스케일에서는 소프트웨어만으로 60fps를 완벽히 유지하기 어렵다.

따라서 NDI는 프리뷰/모니터링 용도로 설계했다. 본방송 송출은 DeckLink SDI가 담당하고, NDI는 감독과 스탭이 각 카메라를 실시간으로 확인하는 데 활용된다. 모든 기술에 동일한 품질 기준을 적용하는 것이 아니라, 용도에 맞게 트레이드오프를 설계하는 것이 중요하다.

Broadcast

DeckLink 4채널 출력에서 Genlock 안 걸면 실제로 어떤 문제가 생기나?

비디오 프레임만의 문제가 아니라, 오디오·LiveLink(모캡)·비디오 — 3종의 싱크가 모두 어긋난다. 라이브 프로덕션에서 가장 까다로운 이슈 중 하나다.

Genlock이 없는 환경에서는 Synology NAS의 NTP로 타임베이스를 통합한 뒤, 오디오는 OBS 단계에서 딜레이를 보정하는 방식으로 우회한다. 오디오 인터페이스에 Genlock이 있으면 하드웨어로 해결되지만, 없는 환경에서 소프트웨어로 동기화를 확보하는 설계 역시 실무에서는 필수적이다.

Infrastructure

AWS EFS로 SVN을 운영했을 때 비용 이슈는?

Unreal 프로젝트 특성상 레벨, 시퀀서 등 바이너리 에셋 용량이 방대하기 때문에, EFS의 GB당 단가가 S3 대비 높아 대용량 바이너리 에셋 저장에는 구조적으로 맞지 않았다.

서브레포 분리를 통한 권한·용량 분산으로 1차 대응하고, 이후 스토리지를 S3로 전환하여 에셋을 분리 수신하는 구조로 변경했다. 스토리지 선택 시 과금 모델과 에셋 특성의 적합성을 사전에 검토하는 것이 중요하다는 교훈이었다.

DevOps

10명 넘는 팀에서 SVN 권한 관리를 어떻게 했나?

직군별 그룹 단위로 설계했다. 배경모델러·애니메이터·라이팅은 시퀀서·맵·프로젝트 관리 권한으로 묶고, 리거·캐릭터모델러·원화가는 프로젝트 관리 권한으로 묶었다. 전반적인 프로세스·엔진 관리·배포는 개발자가 총괄하는 구조다.

핵심 원칙은 권한 경계 = 책임 경계. 직군 간 에셋 접근이 필요한 상황이 발생한다면, 권한을 열어주는 것이 아니라 프로세스 설계를 재검토해야 한다는 신호로 판단한다.

Pipeline

파이프라인 설계 시 가장 중요하게 생각하는 것은?

"사용자가 절대 빌드 시스템을 만지지 않는 구조"를 최우선으로 생각한다. 아티스트 15명에게 Git 명령어를 가르치는 것보다, 원클릭 셋업 스크립트를 만드는 것이 팀 전체의 생산성을 훨씬 더 높인다.

BGRITZSetup을 설계할 때 역할별(Developer/Character/Level/Viewer) 파이프라인을 분리한 이유도 같다. 캐릭터 모델러에게 엔진 소스 빌드가 필요 없고, 레벨 디자이너에게 SVN 전체 체크아웃이 필요 없다. 각 직군이 필요로 하는 최소한의 환경만 자동으로 구성하면, 셋업 시간이 1~2일에서 30분으로 줄어든다.

AI / Automation

AI 자동화를 팀 워크플로우에 어떻게 녹였나?

핵심은 "AI가 해도 되는 반복 작업"과 "사람이 판단해야 하는 작업"을 명확히 분리한 것이다. CLAUDE.md에 레포별 코딩 규칙·커밋 컨벤션·문서 구조를 정의하고, 28개 슬래시 커맨드로 반복 워크플로우를 캡슐화했다.

예를 들어 /done 커맨드 하나로 문서 정리 → 전체 커밋 → Notion 일일보고까지 자동 처리된다. 또한 Notion API를 직접 연동하여 일일/주간 보고서를 자동 생성하고, n8n으로 소품 변형 에셋 파이프라인을 자동화했다. 결과적으로 3개 레포에 걸친 200+ 커밋을 1인이 일관된 품질로 유지할 수 있었다.

Frontend

React + Redux + Electron 런처의 아키텍처 설계 과정은?

IPC 통신 구조, Blob 기반 다운로드 처리, Redux 상태 관리, 카드 구조 UI, 컴포넌트 단위 분리까지 아키텍처부터 직접 설계했다. 다운로드 → ZIP 압축해제 → 설치까지의 파이프라인을 IPC로 연결하고, 진행률을 실시간으로 Renderer 프로세스에 전송하는 구조다.

메인 도메인은 버추얼 프로덕션이지만, 웹·백엔드·인프라 역시 설계부터 운영까지 직접 수행해온 영역이다. 프로젝트에 필요한 기술이라면 도메인을 가리지 않고 풀스택으로 해결한다.
