PBR 파이프라인의 NPR 렌더링 한계

UE5의 Deferred Shading은 PBR 전용. Post-Process Material로는 GBuffer에 커스텀 데이터를 쓸 수 없어 조명 모델 자체를 NPR로 변경 불가.

엔진 소스 직접 수정 — ShadingCommon.ush에 SHADINGMODELID_TOON 추가. Deferred Lighting에서 동작하는 NPR BxDF 구현. Anisotropy로 NPR↔PBR 실시간 블렌딩.

아티스트가 머티리얼 에디터에서 NPR/PBR 비율 즉시 조정. 8개 렌더링 모듈 독립 제어 가능.

Technical Insights

기술 선택의 이유와 설계 철학

왜 Post-Process가 아닌 엔진 소스를 수정했는가

Post-Process Material로도 셀 셰이딩 효과는 낼 수 있다. 하지만 Deferred Shading의 GBuffer에 커스텀 데이터를 기록할 수 없어, 조명 모델 자체를 NPR로 전환하는 것은 불가능하다.

엔진의 Shading Model을 직접 추가하면 Deferred Lighting 단계에서 NPR BxDF가 동작하여, 모든 라이트 소스에 대해 올바른 셀 셰이딩이 구현된다. Post-Process 방식 대비 성능 오버헤드 없이 정확한 라이팅 결과를 얻을 수 있다.

Post-Process 화면 공간 기반 → 조명 정보 접근 불가 → 가짜 셀 셰이딩
Engine Source Mod GBuffer 커스텀 → BxDF 레벨 NPR → 정확한 조명 반응

NDI vs SRT vs WebRTC — 방송 프로토콜 선택

실시간 방송에서 프로토콜 선택은 지연, 화질, 인프라 복잡도의 트레이드오프다.

WebRTC 지연 최소 (~100ms) but 화질 제어 어려움, 시그널링 서버 필요
SRT 원격 전송 최적화, 패킷 손실 복구 but 로컬 환경에서 불필요한 오버헤드
NDI LAN 환경 제로 컨피규레이션, 고화질 저지연, mDNS 자동 디스커버리

스튜디오 환경에서는 NDI가 최적이다. 별도 서버 없이 같은 네트워크의 모든 NDI 디바이스를 자동 발견하고, 비압축에 가까운 화질로 1프레임 이내 지연을 달성한다.

실시간 스레드 안전 — Lock vs Lock-Free 패턴

UE5의 GameThread와 AnimationThread는 병렬 실행된다. 5인 모캡 데이터를 공유할 때 Mutex를 사용하면 한쪽 스레드가 대기하며 프레임 드롭이 발생한다.

TSharedPtr의 Atomic Swap을 활용한 스냅샷 패턴으로 해결했다. 쓰기 스레드가 새 데이터 객체를 만들고 포인터만 원자적으로 교체하면, 읽기 스레드는 이전 객체를 안전하게 계속 사용한다.

Mutex Lock 스레드 대기 → 16.6ms 예산 초과 → 프레임 드롭
Atomic Swap Lock-Free → 대기 없음 → 60fps 안정 유지
