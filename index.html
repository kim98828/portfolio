기술 선택의 이유와 설계 철학

왜 Post-Process가 아닌 엔진 소스를 수정했는가

Post-Process Material로도 셀 셰이딩 효과는 낼 수 있다. 하지만 Deferred Shading의 GBuffer에 커스텀 데이터를 기록할 수 없어, 조명 모델 자체를 NPR로 전환하는 것은 불가능하다고 판단했다.

엔진의 Shading Model을 직접 추가하면 Deferred Lighting 단계에서 NPR BxDF가 동작하여, 모든 라이트 소스에 대해 올바른 셀 셰이딩이 구현되는 편이 낫다고 생각했다. Post-Process 방식 대비 성능 오버헤드 없이 정확한 라이팅 결과를 얻을 수 있을 것이라 판단했다.

Post-Process 화면 공간 기반 → 조명 정보 접근 불가 → 가짜 셀 셰이딩
Engine Source Mod GBuffer 커스텀 → BxDF 레벨 NPR → 정확한 조명 반응

프리뷰 vs 송출 — 프로토콜과 하드웨어의 역할 분리

실시간 방송 시스템에서 "모니터링"과 "본방송 송출"은 요구사항이 다르다고 판단했다. 프리뷰는 유연성과 편의성이 중요하고, 본방송은 프레임 단위의 확실성이 필요하다.

NDI 프리뷰/모니터링 — LAN 자동 디스커버리, 멀티 소스 모자이크, 소프트웨어 기반
DeckLink SDI 본방송 송출 — 하드웨어 확실성, Genlock 프레임 동기화, 방송 표준 충족

NDI는 소프트웨어만으로 동작하여 카메라 전환, 모자이크 프리뷰 등 모니터링 용도에 적합하다고 생각했다. 반면 본방송 송출은 DeckLink SDI 하드웨어를 통해 Genlock 기반 프레임 동기화와 방송 표준을 충족하는 편이 안정적이라 판단했다. 소프트웨어(NDI)와 하드웨어(SDI)의 역할을 분리하면 각각의 강점을 살리면서 시스템 전체의 안정성을 확보할 수 있다.

실시간 스레드 안전 — Lock vs Lock-Free 패턴

UE5의 GameThread와 AnimationThread는 병렬 실행된다. 5인 모캡 데이터를 공유할 때 Mutex를 사용하면 한쪽 스레드가 대기하며 프레임 드롭이 발생할 수 있다고 판단했다.

TSharedPtr의 Atomic Swap을 활용한 스냅샷 패턴이 적합하다고 생각했다. 쓰기 스레드가 새 데이터 객체를 만들고 포인터만 원자적으로 교체하면, 읽기 스레드는 이전 객체를 안전하게 계속 사용한다.

Mutex Lock 스레드 대기 → 16.6ms 예산 초과 → 프레임 드롭
Atomic Swap Lock-Free → 대기 없음 → 60fps 안정 유지

왜 프리컴파일 엔진 배포 전략을 설계했는가

UE5 엔진 소스 빌드는 고사양 PC에서도 40분 이상 소요된다. 캐릭터 모델러, 애니메이터 등 아티스트가 엔진을 직접 빌드하는 것은 비현실적이었다.

해결책으로 역할 기반 배포 전략을 설계했다. 개발자가 커스텀 엔진을 소스에서 빌드하면, 프리컴파일된 바이너리가 SVN NAS로 자동 업로드된다. 아티스트는 SyncPrebuiltEngine 한 번이면 최신 엔진을 받는다.

엔진 소스 수정부에는 // BGRITZ Engine Start YYYY-MM-DD 마커를 의무화하여 커스텀 변경 사항을 추적 가능하게 만들었다. Epic 업스트림 병합 시 충돌 지점을 즉시 파악할 수 있다.

AI 코딩 자동화 — Claude Code를 팀 워크플로우에 통합한 이유

3개 레포(Engine/Setup/Project)에 걸친 작업을 혼자서 관리하려면 반복 작업을 극도로 줄여야 했다. CLAUDE.md로 각 레포의 코딩 규칙을 정의하고, 28개 슬래시 커맨드로 반복 워크플로우를 자동화했다.

/engine-modify는 마커 삽입 + 문서 업데이트를 보장하고, /done은 하루의 마무리(문서 정리 → 전체 커밋 → Notion 일일보고)를 원커맨드로 처리한다. 결과적으로 200커밋을 1인이 3개 레포에 걸쳐 일관된 품질로 유지할 수 있었다.

수동 관리 3개 레포 × 코딩 규칙 + 커밋 컨벤션 + 문서 + 보고 = 컨텍스트 스위칭 과부하

Core Killing Features

XROOM · DNABLE · 공통 — 실전에서 증명한 3가지 핵심 역량

XROOM — Viewport 오버로드 기반 실시간 컴포지팅

실시간 합성(Compositing)을 위해 Viewport를 오버로드하여 전경·렌더·후경을 레이어별로 분리 렌더링했다. 핵심 문제는 3D Plane에 머티리얼을 매핑하면 해상도 저하가 발생한다는 점이었다.

Plane 크기와 정확히 일치하는 위치에 2D로 직접 드로잉하여 3D 경유 없이 원본 화질을 유지하는 방식으로 해결했다. 이 경험을 DNABLE에 그대로 적용하여, 후경의 연기·움직이는 배경을 화질 저하 없이 2D로 렌더링하면서 3D 리소스 비용을 절감했다.

3D Plane 매핑 텍스처 리샘플링 → 화질 저하 → 해상도 의존적
2D Direct Draw Plane 위치 기반 2D 렌더 → 원본 화질 유지 → 3D 비용 제거

DNABLE — Clear Coat 하이잭에서 커스텀 엔진까지

시간 제약 속에서 Clear Coat 채널을 하이잭하여 Toon 렌더링을 먼저 구현하고, 동시에 엔진 소스를 공부하며 배포 전략을 수립했다. 이후 커스텀 엔진으로 전환하여 필요한 버퍼들을 통합하고 우리만의 셰이딩 파이프라인을 완성했다.

상용 커스텀 엔진들의 장점을 분석하여 반음영, 섀도우 채널 분리, 섀도우 블러, 다중 광원, GI, Fake GI를 구현했다. 모든 기능을 성능 저하 없이 매핑하여 실제 송출 테스트까지 완전히 마쳤다.

Phase 1 Clear Coat 하이잭 → 빠른 Toon 프로토타입 + 엔진 학습 병행
Phase 2 커스텀 엔진 → GBufferT + 반음영 + Shadow 채널 + GI → 송출 검증 완료

공통 — 다중 프로토콜 출력과 XR 스튜디오 경험

XROOM과 DNABLE 모두에서 최종 출력을 SDI · NDI · SRT · OSC · WebRTC 등 다양한 포맷으로 동시에 내보내는 시스템을 구현했다. 각 프로토콜의 특성(지연시간, 대역폭, 동기화 방식)을 이해하고 용도에 맞게 라우팅하는 것이 핵심이었다.

이 역량은 XR 스튜디오에서의 실무 경험에서 시작되었다. 가상·실사 합성 환경에서 카메라 트래킹, 타임코드 동기화, 멀티 채널 송출을 직접 운용한 경험이 두 프로젝트 모두에서 프로토콜 설계의 기반이 되었다.

SDI / NDI 방송 표준 송출 + LAN 모니터링 이중화
SRT / WebRTC 인터넷 스트리밍 + 브라우저 실시간 접근
OSC 장비·소프트웨어 간 제어 통합 (카메라, 조명, 모캡)
